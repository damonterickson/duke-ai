## Problem Statement

Army ROTC cadets across the country manually wrestle with spreadsheets and opaque formulas to understand their Order of Merit List (OML) score — the single ranking that determines their branch (job) and duty station for their entire Army career. The OML combines three pillars — Academics (40%), Leadership (40%), and Physical Fitness (20%) — but cadets lack clarity on how individual inputs shift their ranking, spend hours on manual data entry, and miss optimization opportunities because they can't see which pillar offers the highest return on effort for their specific profile.

## Demand Evidence

The founder is the target user — an ROTC cadet living this pain daily. The current workflow involves cobbled-together spreadsheets and manual Accession Sheet tracking. Every cadet in every ROTC battalion in the country faces this same problem. The OML is high-stakes (determines your career), updated infrequently (creating anxiety), and poorly tooled (Excel + guesswork).

## Status Quo

Cadets currently:
1. Receive periodic Accession Sheets from cadre with raw scores
2. Manually enter data into personal spreadsheets or shared battalion Excel files
3. Attempt to calculate projected OML scores using formulas they may not fully understand
4. Have no way to model "what if" scenarios without rebuilding spreadsheet logic
5. Get generic advice from cadre ("improve your PT score") with no personalized prioritization
6. Have zero visibility into how they compare nationally until official rankings drop

Cost: hours of manual tracking, missed optimization opportunities during the critical MSIII year, and anxiety from operating blind on career-determining decisions.

## Target User & Narrowest Wedge

**Primary:** MSIII cadets in their ranking year — they have the most to gain and the highest urgency.
**Secondary:** MSI/MSII cadets building early, MSIV cadets tracking final standings.

The wedge is the AI mentor itself. A cadet opens the app, tells Vanguard AI their current scores, and immediately gets: "Here's your projected OML. Your biggest opportunity is your 2-mile run — dropping 30 seconds adds more points than raising your GPA by 0.2. Here's a 4-week plan." That first interaction is the hook.

## Constraints

- **Expo React Native** — cross-platform mobile (iOS + Android), already scaffolded
- **Offline capability needed** — field training, barracks with poor connectivity
- **OML formula must be configurable** — while stable year-to-year, scoring weights and component calculations should not be hard-coded as constants
- **Data sensitivity** — cadet performance data is personal; local-first storage, no mandatory cloud sync
- **Military aesthetic** — "Modern Military" design language (olive drab, gold accents, high-contrast, authoritative)

## Premises

1. The OML formula is stable enough to encode but should be configurable for year-to-year adjustments.
2. Cadets will manually enter data initially — Canvas/OCR integrations are enhancement layers, not prerequisites.
3. The AI advisor's value is personalized optimization, not generic advice.
4. Offline/low-connectivity support is essential.
5. Adoption is peer-driven, not top-down mandated.

## Approaches Considered

### Approach A: Layered Launch
Build all features in 3 sequential waves. Wave 1: calculator + dashboard. Wave 2: AI + simulator. Wave 3: Canvas + OCR. Each wave is independently useful.
- **Effort:** L (human: ~3 months / CC: ~4 weeks)
- **Risk:** Low
- **Pros:** Early feedback, incremental value, lower integration risk
- **Cons:** Dashboard-first means the AI feels bolted on rather than central

### Approach B: Full Send
Build everything simultaneously, ship when complete.
- **Effort:** XL (human: ~5 months / CC: ~6 weeks)
- **Risk:** High — long build, compounding integration bugs, no user feedback loop
- **Pros:** Complete vision from day one
- **Cons:** High risk of building the wrong thing; long time to first user

### Approach C: AI-First Architecture (CHOSEN)
The AI mentor (Vanguard AI) is the core experience. The app opens to a chat/advisory interface, not a dashboard. Everything else — data entry, score tracking, fitness logs — exists to feed the AI better data so its recommendations improve. The dashboard is a secondary view the AI can reference, not the primary UI.
- **Effort:** L (human: ~3 months / CC: ~4 weeks)
- **Risk:** Medium — AI quality depends on data quality; conversational UX requires more design thought
- **Pros:** Differentiated from "another calculator app"; the mentor relationship creates retention; naturally encourages data entry (AI gets smarter as you feed it); feels like having a knowledgeable senior cadet in your pocket
- **Cons:** Cold start problem (AI needs data to be useful); chat UX is harder than forms; LLM API costs

## Recommended Approach

**Approach C: AI-First Architecture.**

The core insight: a calculator is a commodity — anyone can build an OML spreadsheet. What cadets actually need is a *mentor* who knows their specific profile and tells them what to do next. The AI is the product. Everything else is infrastructure that makes the AI smarter.

### Architecture

```
┌─────────────────────────────────────┐
│         Vanguard AI (Chat UI)       │  ← Primary interface
│  "What should I focus on this week?"│
├─────────────────────────────────────┤
│        Context Engine               │  ← Assembles cadet profile for LLM
│  OML scores, goals, history, trends │
├──────────┬──────────┬───────────────┤
│ Score    │ Fitness  │ Leadership    │  ← Data layers (feed the AI)
│ Engine   │ Log      │ Log           │
├──────────┴──────────┴───────────────┤
│  MMKV (settings/prefs) + SQLite     │  ← Offline-first
│  (structured score history/trends)  │
└─────────────────────────────────────┘
(Canvas Sync + OCR Upload are post-v1 — not in initial architecture)
```

### Context Engine

The Context Engine assembles the cadet's full profile into a structured prompt for the LLM. It is the bridge between local data and AI intelligence.

**Fields included in every LLM call:**
- Cadet profile: year group (MSI-MSIV), battalion, target branch, goal OML score
- Current pillar scores: Academics (GPA, credits, MSL course grades), Leadership (eval scores, command roles, CST performance), Physical (AFT total + event breakdown)
- Computed OML: current projected score, pillar sub-scores, delta since last entry
- Trend data: last 3 data points per pillar (or all available if fewer)
- Last 5 conversation turns (for continuity)

**Format:** Structured JSON block injected into the system prompt, followed by a natural language summary. Example:
```json
{"gpa": 3.45, "msl_gpa": 3.67, "aft_total": 560, "aft_events": {"spt": 10.2, "hrp": 52, "sdc": "1:48", "plk": "2:15", "2mr": "14:30", "alt_event": null}, "leadership_eval": 78, "cst_score": 82, "command_roles": ["Squad Leader"], "projected_oml": 842, "pillar_scores": {"academic": 336, "leadership": 312, "physical": 194}, "pillar_deltas": {"academic": "+4", "leadership": "0", "physical": "+8"}, "target_branch": "Infantry", "year_group": "MSIII"}
```

**Token budget:** ~800 tokens for context, leaving the majority of the context window for conversation. If history exceeds budget, oldest conversation turns are trimmed first, then trend data is summarized.

**System prompt content:** The system prompt must include:
- **OML formula overview:** Three pillars, their weights (40/40/20), how sub-scores roll up to the total
- **Optimization heuristics:** Marginal OML gain per unit improvement per pillar (e.g., "1 point of AFT improvement = X OML points vs. 0.1 GPA increase = Y OML points") — these are computed by the Score Engine and injected dynamically
- **Branch target ranges:** Known competitive OML score ranges by branch (e.g., Infantry ~750+, Aviation ~850+) — sourced from historical data or cadet community knowledge
- **Persona instructions:** "You are Vanguard AI, a knowledgeable senior cadet and OML mentor. Be direct, specific, and actionable. Never give generic advice. Always reference the cadet's actual numbers. Speak in a tone that respects military culture — professional but not stiff."
- **Guardrails:** Do not speculate about other cadets' scores. Do not guarantee rankings. Always caveat projections as estimates.

### Cold Start Flow

On first launch (zero data), the AI does not present an empty chat. Instead:

1. **Welcome screen:** "I'm Vanguard AI — your OML mentor. Let's get your baseline. This takes about 2 minutes."
2. **Guided onboarding:** 5-screen flow collecting: year group, current GPA, most recent AFT total (or individual events), any leadership roles held, target branch.
3. **First insight:** Immediately after onboarding, the AI delivers a personalized briefing: "Based on what you've shared, here's your estimated OML and what we know about your opportunities. Add your leadership eval scores to unlock a full recommendation." Frame data incompleteness as progressive disclosure — the AI gets smarter as the cadet provides more data.
4. **Chat unlocks:** Full conversational AI is now available with baseline context.

Minimum viable data for AI to function: year group + GPA + AFT total. All other fields enhance recommendations but are not required.

### Offline Mode

The app has three connectivity tiers:

| Feature | Online | Offline |
|---------|--------|---------|
| OML calculator & what-if sim | Full | Full (runs locally) |
| Data entry | Full | Full (writes to local DB) |
| Dashboard & trends | Full | Full (reads local DB) |
| AI chat | Full (live LLM) | Degraded (see below) |

**Offline AI behavior:**
- On each successful AI response, the app caches: the daily briefing, the most recent optimization recommendation, and the last what-if result.
- When offline, the chat shows: "Vanguard AI is offline. Here's your last briefing from [date]:" followed by the cached content.
- The what-if simulator works fully offline (pure math, no LLM needed).
- **Offline query handling (Wave 1):** If a cadet types a question while offline, the app shows: "I can't answer that offline, but I've saved your question. I'll respond when you're back online." The question is persisted to SQLite. On reconnect, queued questions are sent and responses appear as new messages.
- Cache uses MMKV (small key-value pairs). No eviction needed — only latest per category is stored.

**LLM API error handling:** On API errors (rate limits, service outage, billing exhaustion, or daily token cap), the chat displays: "Vanguard AI is temporarily unavailable." and falls back to the cached briefing. The error is logged locally for debugging. The what-if simulator and OML calculator continue to work (no LLM dependency).

### Data Integrity

Every data entry field enforces validation at input:

| Field | Type | Valid Range | Format |
|-------|------|-------------|--------|
| GPA | decimal | 0.00–4.00 | 2 decimal places |
| AFT Total | integer | 0–600 | auto-calculated from events |
| SPT (Standing Power Throw) | decimal | 0.0–15.0 | meters, 1 decimal |
| HRP (Hand-Release Push-ups) | integer | 0–100 | count |
| SDC (Sprint-Drag-Carry) | time | 0:00–5:00 | mm:ss |
| PLK (Plank) | time | 0:00–5:00 | mm:ss |
| 2MR (2-Mile Run) | time | 10:00–25:00 | mm:ss |
| Leg Tuck | integer | 0–25 | count (alternate event) |
| Leadership Eval | integer | 0–100 | Commander's Assessment score |
| MSL Course Grade | letter | A–F | standard letter grade |

Invalid input is rejected inline with a clear error message. The OML engine refuses to calculate if any required field has an invalid or missing value — it shows which fields need attention.

**AFT scoring tables** (raw performance → scaled points, by age bracket and gender) must be modeled as configurable lookup data (JSON files), not hard-coded logic. This is a blocking dependency for the OML engine — the tables must be extracted from **Army FM 7-22 (Holistic Health and Fitness)**, not DA PAM 611-21 (which covers branch management, not fitness scoring). Use USACC Reg 145-1 for the OML formula and pillar weights.

**AFT alternate events:** The ACFT allows one alternate event substitution (e.g., Leg Tuck instead of Plank). The OML engine must accept either the standard 6-event set or one valid alternate substitution. In the data model, the alternate event replaces its standard counterpart — the replaced event is stored as `null`. Scoring tables for each alternate must be loaded from the same configurable JSON. MSL course grades are converted to grade points (A=4.0, B=3.0, C=2.0, D=1.0, F=0.0) and averaged separately as the MSL GPA sub-score, which feeds into the Academics pillar calculation per USACC Reg 145-1.

### Key Screens

1. **Vanguard AI (Home)** — Chat interface. Proactive daily briefing generated on app foreground if the last briefing is >20 hours old. If no new data since last briefing, the cached version is shown without a new LLM call. ("Good morning, Cadet. Your projected OML is 842. Here's what I'd focus on today.") Cadets can ask questions, run what-if scenarios, or get optimization recommendations.

2. **Mission Profile (Dashboard)** — Secondary view. Pillar breakdown (Academics 40%, Leadership 40%, Physical 20%), trend lines, projected score. The AI references this data but the cadet can also browse it directly.

3. **Data Entry Flows** — Guided input screens for: GPA/course grades, AFT scores (event-by-event), leadership evaluations (Commander's Assessment 0-100), extracurriculars. Each entry triggers an AI insight ("Nice — that 580 AFT pushed you up ~12 OML points.").

4. **What-If Simulator** — In Wave 1, available via natural language through chat ("What if I get an A in MSL 301?"). In Wave 2, gets a standalone slider UI with real-time OML projection.

5. **Settings & Profile** — Year group, battalion, target branch, goal OML score. Feeds AI personalization.

### Tech Stack

- **Framework:** Expo React Native (already scaffolded)
- **Navigation:** Expo Router (file-based routing)
- **State/Storage:** Zustand (runtime state) + MMKV (settings, preferences, AI cache) + SQLite via expo-sqlite (structured score history, trend data, multi-entry logs)
- **AI:** Claude API (Anthropic) or OpenAI — system prompt with OML domain knowledge, cadet profile injected as context
- **OML Engine:** Pure TypeScript module — deterministic score calculation, no AI needed for math
- **UI Components:** Custom component library with "Modern Military" design tokens
- **Charts:** react-native-chart-kit or Victory Native for trend visualization

### Build Sequence

**Wave 1 (Week 1-2): Core Loop**
- OML calculation engine (TypeScript, fully tested, AFT scoring tables as configurable JSON)
- Local data storage (SQLite for scores/history, MMKV for settings/cache)
- Cold start onboarding flow (5-screen guided data entry)
- Basic data entry screens with validation (GPA, AFT scores, leadership evals)
- Vanguard AI chat interface (connected to LLM API)

**Wave 2 (Week 2-3): Intelligence Layer**
- What-if simulator standalone slider UI
- Personalized optimization recommendations (pillar-level ROI analysis)
- Mission Profile dashboard with pillar breakdown
- Trend tracking over time (SQLite queries → charts)
- AFT event-by-event scoring with age/gender bracket tables
- Leadership & extracurricular logging screens

**Wave 3 (Week 4+): Enhancement Layer — Post-Pilot**
- Canvas LMS integration (grade sync) — requires institutional API access

**Deferred (Post-v1):**

## Open Questions

1. **LLM Provider:** Claude API vs OpenAI? Key factors: quality of domain-specific reasoning, cost at scale, rate limits. Recommend starting with Claude Sonnet for chat + Haiku for micro-insights.
2. **OML Formula Source:** Is there an authoritative, published OML formula, or does it vary by year/source? Need to validate the calculation engine against official Cadet Command documentation. **Risk note:** If the published formula is incomplete or has undocumented edge cases (e.g., handling of incomplete semesters, transfer credits, medical profile exemptions), the OML engine will be built against best available documentation. Success criterion should be amended to "matches within ±1% for standard profiles; known edge cases documented."
3. **Canvas API Access:** Does your university's Canvas instance allow student-level API access, or does this require institutional approval? (Deferred to post-v1.)
4. **Data Privacy:** Local-only for v1. Cross-device sync is a post-pilot consideration that requires a privacy architecture review.
5. **Leadership Sub-Weights:** The exact breakdown of the Leadership pillar (40%) into sub-components (Commander's Assessment, CST, CLC, command positions, extracurriculars) needs validation against official documentation. This is a blocking unknown for accurate OML calculation.
